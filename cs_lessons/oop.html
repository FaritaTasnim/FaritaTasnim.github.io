Object-Oriented Programming

What's the most efficient way to write code meant to generate a lot of different objects that have overlapping sets of common properties and functions?


- CHAPTER 1: Templating Code -



In the revolution to overthrow the tyranny of The Ampere, Princess Leila is designing a set of 5000 droids to help the efforts of the Capacitance to restore balance to the galaxy.



All of the droids have the same set of basic functions they must perform. Some examples include that they must be able to explore any planetary terrain, operate a spaceship, navigate space, and interpret human language.

However, instead of making all of the droids the exact same, Leila wants to create different types of droids that can also do specific sets of tasks, which are more specialized functions than the basic ones. For example, she needs spaceship mechanic droids, medical droids, chef droids, and assassin droids. Let's say she has 20 different specific types of droids she wants to create.

Leila is coding up the design for her robots, but she has many other things on her to do list. Accordingly, she needs to code the properties and functions for each of her droids in the most efficient way possible.

Which of the following strategies would be most efficient, so that Leila has to spend the least amount of time coding? Hint: The goal is for her to write code so that she minimizes the number of lines of repeated code.



A. Write a new chunk of code (~300 lines) specifying the properties and functions of every single droid in her droid set, and invoke (with a single line) one of those chunks of code to create each droid in her droid set



B. Write a new chunk of code (~300 lines) specifying the properties and functions of every type of droid (mechanic, medic, chef, assassin, etc.), and invoke (with a single line) one of those chunks of code to create each droid in her droid set.



* C. Write a chunk of code specifying the droid with the very basic properties (~100 lines) and functions that all droids possess, extend that code into a new chunk of code (~200 lines) for each different type of droid (mechanic, medic, chef, assassin, etc.), and invoke (single line) one of those chunks of code to create each droid in her droid set



Why is the correct answer C and not B? Because in B, even though you're efficient in the sense that you only have large code chunks for each type of droid, you must repeat in each of those chunks the code for the basic properties and functions common to all droids, no matter their specific purpose.

This is a simple idea. Create templates (chunks of code) and reuse them multiple times when you need to create lots of computing objects with the same set of properties and functions! And this simple concept is at the heart of one of the most powerful programming practices, referred to as object-oriented programming. When certain properties and functions must be present in a large set of many computing objects, we create one of these templates, which is referred to as a class.


- CHAPTER 2: Inheritance -



Of course, as in Leila's droid situation above, computer scientists often want to create many different templates, and sometimes even the templates themselves share some common properties. In this case it is most efficient, as you answered above, to create one master template (called a parent class or super class) and then extend the template to more specific templates (called child classes or subordinate classes). Each child class that extends a parent class inherits the properties and functions of the parent class, so that those properties and functions don't have to be coded again. However in general, the parent class does not obtain any of the properties or functions of any of its child classes.

This concept of inheritance can be drawn in terms of a tree diagram. Here's what Leila's code would look like as a tree diagram. We've only depicted 4 of the 20 "child" droid classes below.



With this kind of class hierarchy, each of the droids (each referred to as a separate object) can be created (initialized) with a single line of code invoking (calling) the class to which it belongs. In this single line, any properties specific to each droid, such as droid_id, will be provided.

Medic_791 = Medic(droid_id = "medic_791")

Leila thought a little more and realized that even some of these child classes would share some common properties and functions. She realized she has a set of math/science droids, a set of engineering/mechanic droids, a set of human interaction/service droids, and a set of fighting droids.

This concept of inheritance is pretty powerful because it can involve multiple levels! This allows us to reuse lots of code without having to rewrite that code over and over.

- CHAPTER 3: Composition (Grouping) -



Leila thinks more and realizes she could make her code even more modular. She realizes that all of her droids have some set of the same core components, and that some of the more specific types of droids have more specific shared components, and so on.

She wants her code for droid components to be separate from the code for her droids. This is possible with object-oriented programming! She can create a separate class structure for droid components and code her droid classes to have instances (objects) of different classes specifying droid components. This is called composition.

Let's say all droids have one central processing unit, one visual processing unit, one audio processing unit, and one chemical processing unit.

For movement, a droid could have any of three different types of movement units (limb, wheel, jetpack). Math/science droids have two limbs and two wheels. Engineer/mechanic droids have two limbs and two wheels. The space mechanic droids each have a jetpack and two wheels. Human interaction/service droids have four limbs. Fighting droids have two wheels and a jetpack.

There are many other types of components but for now let's sort out the above.

You can help Leila by drawing arrows from each component to the droid class that should contain an instance of it. Draw multiple arrows if a droid has multiple of the same component. To draw an arrow, click on the two classes the arrow connects. The arrow should point from one class toward the class that contains it.



Nice! So you see that by allowing a class to consist of instances of other classes, composition allows for the relationship structure among shared features and properties to be more like an arbitrary network instead of a strict tree-like structure.

In programming lingo, this distinction between inheritance and composition is commonly described as: Inheritance provides relationships of the style is-a, whereas composition provides relationships of the style has-a.


- CHAPTER 4: Aggregation (Abstract Groups) -



As the final feature in Leila's design of her large set of droids, she wants groups of droids to be able to work together. In particular, she wants to be able to define different groups of droids so that together, they have certain collective properties and functions.

For example, she wants to be able to create a group of fighter pilot droids for a battle mission so that they can collectively orient themselves into different formations for attack and defense during a space battle with the Ampere.

What is a strategy Leila could use in her code to make this possible in the most efficient way?

Create instances of fighter pilot droids and write non object-oriented code for getting them to coordinate in different ways.

* Create a new class, such as Attack_Group, that contains instances of fighter pilot droids, and contains functions that allow for different types of coordination. Then she can create an instance of an Attack_Group when she wants to go on a battle mission, and call the functions for that group as desired.

Awesome, so Leila can use object-oriented programming to make coding groups of things more efficient, just as we saw before with the droid components.

However, there's one issue... Once the mission is over, Leila wants the group of fighter pilot droid to disband and go back to being used for other fighting tasks and potentially to be used in different groups. Composition doesn't allow for something like that. Deleting the a composition deletes all objects it contains.

Instead, we can do what's called aggregation. For abstract collections of entities, instead of instantiating droid objects inside the class for the group, we instantiate the droid objects separately, and refer to them when we instantiate a group class.

Here's what creating a new aggregation might look like, using a set of droid objects that have already been instantiated elsewhere:

New_Attack_Group = Attack_Group(attack_droids = {Fighter_Pilot_007, Fighter_Pilot_489, Fighter_Pilot_320, Fighter_Pilot_157, Fighter_Pilot_294, Fighter_Pilot_287, Fighter_Pilot_187})

All of these practices of creating network-type relations between different pieces of code --- inheritance, composition, and aggregation --- makes object-oriented programming a very powerful paradigm.

In programming, this kind of power is referred to as polymorphism (poly = many, morph = form), which is just a fancy word saying that there's coherent network of code that could create many different types (forms) of things.
